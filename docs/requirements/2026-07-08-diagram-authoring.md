# 要求: 図の作成フローと管理表現（2026-07-08 追補）

日付: 2026-07-08。不変の要求スナップショット。
[2026-07-08 generic-pptx-walker 要求](2026-07-08-generic-pptx-walker.md) の
追補であり、置き換えではない。ユースケース節（同 §UC）を図の観点で具体化する。

## 1. 背景（ユーザー提供の一次情報）

最も頻度の高いユースケースは、deck の「GitHub と開発ライフサイクル」
ページのような**密なアーキテクチャ図・ライフサイクル図の構成**である。
現物は Diag DSL（絶対座標の Vue コンポーネント約 90 要素）で
deck.md に直書きされている（tech-slide slides/deck.md:95-）。

図の作成フローは 3 系統ある:

- **UC-D1（再構成）**: 既存の pptx から図を抜き出し、意味論を理解した
  うえで再構成する。
- **UC-D2（対話構成）**: ユーザーが Coding Agent とインタラクティブに
  図を構成する。GitHub Copilot App の canvas のような**双方向 UX**
  （agent/ユーザー双方が同じ図を見て操作する）が望ましい。
- **UC-D3（手書き）**: ユーザーがソースを直接書く。

これを実現するうえで**図の管理表現（正準表現）が最重要**であり、
Slidev への組み込みと pptx 化はそれに整合させる。

## 2. 現状分析

現行表現 = deck.md 埋め込みの Diag markup。評価:

| フロー | 成立度 | 理由 |
|---|---|---|
| UC-D1 | 低 | pptx からの抽出結果を「markdown 内 Vue markup」として生成・検証するのは困難 |
| UC-D2 | 半分 | agent→表示は Slidev HMR で既に成立。表示→ソース（canvas 操作の書き戻し）は markdown 埋め込み markup が対象では困難 |
| UC-D3 | 高 | 現に運用中。ただし座標手調整が重い |

構造的な観察: Diag DSL は実質すでに宣言的データだが、
「markdown 埋め込みテキスト」という搬送形式に閉じているため、
スキーマ検証・プログラム的読み書き・複数ツールでの共有ができない。

## 3. 追加要求

- **FR-D1（正準モデル）**: 図 1 枚は 1 つのスキーマ付きモデルファイルで
  管理できること。deck からは参照で読み込む。モデルは既存 Diag 語彙
  （box/edge/icon/text/chevron/…、fill トークン、テーマ変数参照）を
  そのまま写像した形とし、表示・agent 編集・canvas 編集・pptx 抽出・
  pptx 生成のすべてが同一モデルを共有する。
- **FR-D2（双方向編集）**: モデルファイルへの書き込みが表示に即時反映
  される（Slidev HMR）。canvas 側の操作はモデルファイルへの書き戻しとして
  実現し、「canvas が真実を持つ」状態を作らない。ファイルが唯一の真実。
- **FR-D3（pptx 抽出）**: pptx の native shape 群を本モデルへ写像する
  抽出経路を持てること（UC-D1）。抽出は完全自動である必要はなく、
  「候補モデル + 人間/agent による意味論の付与」の二段でよい。
- **FR-D4（モデル直変換の余地）**: 図に限っては、DOM 実測を経ない
  「モデル → PptxGenJS 直変換」経路を選べる余地を残すこと。
  構造化データ入力なら native 忠実度が出せることは Figma プラグイン
  調査（landscape 2026-07-08 §Figma）で確認済み。
- **FR-D5（escape hatch）**: 現 DSL が Vue の表現力に逃げている箇所
  （例: DiagBox 子スロットのインライン img 列）について、モデル側で
  語彙として吸収するか、明示の escape hatch（Vue 断片の埋め込み）を
  許すかを決めること。escape hatch を許す場合、その部分は canvas 編集・
  モデル直変換の対象外になることを仕様として明記する。
- **NFR-D1（agent 編集性）**: モデルはテキスト diff が意味を持つ形式
  （YAML/JSON）とし、スキーマにより agent の生成物を機械検証できること。
- **NFR-D2（後方互換）**: 既存の deck.md 直書き Diag markup は引き続き
  動作すること。モデルファイル化は追加であり置き換えではない。

## 4. トレードオフの認知

- モデル化により Vue の自由表現（任意スロット・任意 style）は制限される。
  制限こそが canvas/抽出/直変換を可能にする（表現力と道具立ての交換）。
- 既存エディタ形式（draw.io / Excalidraw / tldraw）を正準にする案は、
  エディタ実装を無償で得られる代わりに、テーマ変数継承・Diag 語彙との
  対応・pptx 語彙への写像で劣り、agent にとっても冗長な形式になる。
  正準は自前スキーマとし、必要ならそれらエディタとの変換を周辺に置く。
- 正準表現の決定は ADR-0018 で行う。
